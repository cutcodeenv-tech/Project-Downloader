# Газеты: импорт статей — установка на новой машине и известные проблемы

Новая функция (вкладка «Газеты» → блок «Импорт статей») тянет за собой зависимости,
которых раньше в проекте не было. Ничего из этого не ставится через `requirements.txt` —
это отдельный, Node.js-based пайплайн (`paper_pipeline/`).

## Что нужно поставить на новой машине

1. **Node.js** (тестировалось на v24.12.0, подойдёт любой недавний LTS). Раньше в проекте
   Node вообще не использовался — теперь это обязательная системная зависимость только
   для этой функции. Добавил проверку в `check_deps()` (`web/app.py`) — появится строка
   «Node.js» на вкладке «Зависимости»; если её нет — `brew install node` (macOS).
2. **Google Chrome** должен стоять по стандартному пути
   `/Applications/Google Chrome.app/Contents/MacOS/Google Chrome` — `shot.mjs` обращается
   к нему напрямую (puppeteer-core, без своего браузера). Тоже добавил в проверку
   зависимостей. Только macOS — на другой ОС этот путь в `shot.mjs` (`const CHROME = ...`)
   надо будет поменять вручную.
3. **`ffmpeg`** — уже было требованием проекта, ничего нового, но используется и здесь
   (сборка макета страницы, рендер маски).
4. **npm-зависимости самого пайплайна** — их надо поставить один раз на каждой машине:
   ```bash
   cd paper_pipeline
   npm install
   ```
   `node_modules/` теперь в `.gitignore` (раньше там вообще не было записи про
   node_modules — если бы не добавил, `git add paper_pipeline` закоммитил бы весь npm-пакет).
   Ставит `puppeteer-core` — чистый JS, без бинарных модулей, так что от архитектуры
   машины (Intel/Apple Silicon) не зависит.
5. **hyperframes CLI** — НЕ зафиксирован по версии. Все вызовы — это `npx --yes hyperframes
   ...` без версии, то есть при каждом первом запуске на новой машине npx скачает
   актуальную версию с npm (нужен интернет на первый прогон, дальше кэшируется). Тестировал
   на 0.7.62. Если это важно (воспроизводимость между машинами), можно захардкодить
   версию, например `hyperframes@0.7.62`, во всех трёх местах, где вызывается npx, в
   `shot.mjs`/`compose.mjs`/`web_generate.mjs` — сейчас так не сделал, чтобы не
   рассинхронизироваться с остальным пайплайном (Websites animate), где так же не
   зафиксировано в большинстве внутренних вызовов.

## Известные проблемы (нужно чинить руками)

### 1. Генерация маски «с нуля» — сломана
Если удалить `paper_pipeline/footages/paper-mask-N.png` (или `paper-geom-N.json`) для
какого-то шаблона, `compose.mjs` пытается перегенерировать её сам (рендерит временную
композицию через `npx hyperframes render ... --format png-sequence`) — и падает с
ошибкой `mask is empty — template render failed`. Поймал это вживую при первом тесте на
этом проекте. Причину не диагностировал (похоже на проблему конкретно с этим
временным рендером/версией hyperframes CLI — та же логика в оригинальном проекте
`Websites animate/Paper` тоже никогда не тестировалась «с нуля», там всегда были готовые
маски).

**Обход**: скопировал уже готовые `paper-mask-{2,3,4,5}.png`, `paper-mask-{2,3,4,5}-hard.png`,
`paper-geom-{2,3,4,5}.json` из `Websites animate/Paper/footages/` — они валидны, потому что
`data{2,3,4,5}.json` в обоих проектах побайтово одинаковые. **Не удаляйте эти файлы** —
без них шаблон, для которого удалили кэш, перестанет собираться.

### 2. `.venv` — архитектура/версия Python (проблема ЭТОЙ машины, не самого кода)
Обнаружил и частично починил на этой (Extreme SSD) машине во время тестирования — но если
переносите проект физическим копированием (а не через `git clone`), эта же поломка
может повториться на новой машине:
- Симлинк `.venv/bin/python3.11` указывал на `/usr/local/opt/python@3.11/...`
  (Intel-путь Homebrew) — на Apple Silicon Homebrew стоит в `/opt/homebrew`, симлинк был
  битый. Починил (`ln -sf /opt/homebrew/bin/python3.11 .venv/bin/python3.11`).
- Но пакеты в `.venv` (torch и т.д., для whisper) собраны под x86_64 — на arm64 падают
  с `OSError: ... incompatible architecture`.
- **`.venv/` в `.gitignore`, так что при `git clone` на новой машине эта проблема не
  возникнет — venv соберётся заново под её архитектуру через `./start.sh web`.** Проблема
  актуальна только если копируете папку проекта целиком (внешний диск между Mac'ами).
  Если после переноса `./start.sh web` падает на импорте torch/whisper с ошибкой
  архитектуры — удалите `.venv/` и запустите `./start.sh web` заново, чтобы пересобралось
  под нужную архитектуру.

### 3. CSV парсится по регулярке, не по колонкам
`web_generate.mjs` вытаскивает из CSV любые подстроки `https?://...` регуляркой — не
учитывает кавычки/колонки. Если в URL внутри CSV-поля в кавычках есть запятая
(`...?x=1,2`), хвост после запятой обрежется. Для реальных новостных ссылок это редкость,
но если понадобится точный CSV-парсинг — сейчас его нет.

### 4. Нерусские статьи — пропускаются, не переводятся
Если заголовок статьи не на кириллице, `web_generate.mjs` просто пропускает её
(лог: «пропущено — текст не на русском») и идёт дальше по списку. Автоматического
перевода в веб-версии нет (в отличие от интерактивной работы с ИИ-агентом, где перевод
можно сделать вручную, отредактировав `content.json`).

### 5. Рендер на этом внешнем SSD — медленный
Один 9-секундный ролик рендерился ~14 минут на этом диске (для сравнения — на внутреннем
диске в тестовом прогоне уходила ~1 минута). Похоже на упор в скорость записи/чтения
внешнего SSD. Для пачки статей через CSV это может ощутимо накопиться — если станет
проблемой, стоит подумать про рендер во временную папку на внутреннем диске с
последующим переносом готового файла.

### 6. Один общий слот задачи
Кнопки «Сгенерировать»/CSV используют тот же `_start_job`/SSE-механизм, что и все
остальные действия в приложении — одновременно может выполняться только одна задача на
всё приложение (как и везде в проекте, это не новое ограничение, просто относится и сюда).

## Что уже сделано и не требует повторных действий

- `paper_pipeline/` — весь код и ассеты (шрифты, footages, кэш масок, data2-5.json)
  уже в проекте, кроме `node_modules/` и `renders/` (гитигнорены).
- `web/app.py` — два новых роута (`/api/gazety/articles/generate`,
  `/api/gazety/articles/upload_csv`) + проверки Node.js/Chrome в `check_deps()`.
- `web/templates/index.html` — блок «Импорт статей» на вкладке «Газеты».
- `.gitignore` — добавлены `node_modules/`, `paper_pipeline/renders/`,
  `paper_pipeline/.rotation`.

## Быстрая проверка после установки (без браузера, из терминала)

```bash
cd paper_pipeline
node web_generate.mjs --urls-json '["<ссылка на русскоязычную статью>"]' --out-dir /tmp/test_out
```
Если в `/tmp/test_out` появился `.mov` (ProRes 4444, альфа-канал, `yuva444p12le`) — пайплайн
настроен верно, можно запускать через веб-интерфейс.
